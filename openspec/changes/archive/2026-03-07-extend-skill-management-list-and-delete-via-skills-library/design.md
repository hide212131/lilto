## Context

現在の Skill 管理は `skills` ライブラリ経由のインストールを中心に実装されている一方、一覧・削除はアプリ内ファイル探索ベースで運用されており、責務境界が混在している。加えて、配布形態を考慮した実行方式（同梱ランタイム）や symlink スキル検出など、実装上の重要判断が artifacts で明文化されていない。

今回の変更では、既存実装を後付けで仕様化しつつ、今後の `skills` ライブラリ活用（一覧・削除）を前提に、Main/IPC/Renderer のインターフェースを一貫化する。

## Current State Audit (2026-03-07)

- `install` は `installSkillFromSource()` で `skills` CLI（`skills add`）を実行している。
- `list` は `listSkillsWithSource()` が `discoverSkillMetadata()` を使って `userSkillsDir` / `bundledSkillsDir` を直接走査している。
- `remove` は `uninstallUserSkill()` が path 境界チェック後に `fs.rmSync()` で直接削除している。
- つまり現状は「追加のみ `skills` ライブラリ経由、一覧/削除はファイルシステム主体」で管理責務が分離している。

## Goals / Non-Goals

**Goals:**
- Skill 管理（インストール・一覧・削除）の責務を仕様として整理し、OpenSpec 上で追跡可能にする。
- 一覧・削除を `skills` ライブラリベースへ拡張するための要件を確定する。
- OAuth 利用時でも Skill 管理 UI とランタイム反映が一貫して動作する前提を定義する。
- 配布環境での実行を考慮し、外部 `node` / `npx` 非依存の実行要件を明示する。

**Non-Goals:**
- この change で `skills` ライブラリ内部実装を変更しない。
- Skill の審査・セキュリティスコアリングロジックをアプリ側で再実装しない。
- 新しい Skill 形式（SKILL.md 以外）を追加しない。

## Decisions

1. **Skill 管理 API を「操作単位」で分離する**
   - インストール・一覧・削除をそれぞれ独立した要件として定義する。
   - 理由: 既存は「インストール成功」中心で、一覧/削除の振る舞いが暗黙的になっているため。
   - 代替案: 単一要件に集約する。
   - 不採用理由: 変更点の追跡粒度が粗く、将来の差分検証が困難。

2. **一覧・削除は `skills` ライブラリとの整合を優先する**
   - アプリ独自探索結果のみを正とせず、`skills` ライブラリの管理状態と矛盾しない仕様にする。
   - 理由: インストール実行元が `skills` である以上、管理面も同一ソースに寄せる方が整合性が高い。
   - 代替案: 現行のファイル探索を継続。
   - 不採用理由: symlink・lock file・配布時の実パス差異で不整合が起きやすい。

3. **反映タイミングは「次回送信から有効」を基準とする**
   - Skill 管理操作成功時にランタイム側セッションを無効化し、次回実行で再初期化する。
   - 理由: アプリ再起動不要で UX を維持しつつ、状態不整合を最小化できる。
   - 代替案: 常時ホットリロード。
   - 不採用理由: セッション中断やツール状態維持との競合リスクが高い。

4. **配布実行は同梱ランタイム前提とする**
   - subprocess 実行は `process.execPath` を使用し、必要時 `ELECTRON_RUN_AS_NODE=1` を付与する。
   - 理由: 配布先のシステム `node` 依存を避けるため。
   - 代替案: `node` / `npx` を直接実行。
   - 不採用理由: 配布先環境依存で失敗率が高い。

5. **一覧・削除の user skill 管理責務は `skills` ライブラリへ寄せる**
   - user skill の一覧・削除は `skills` ライブラリが管理する状態を優先ソースにする。
   - 一覧結果は「`skills` ライブラリ管理の user skill」+「アプリ固有 bundled skill」の和集合として返す。
   - bundled skill はアプリ管理資産として削除対象外（MUST NOT）を維持する。
   - 理由: 追加経路と同じ管理境界へ揃えることで、symlink/実体差異や lock/state 不整合を減らすため。
   - 代替案: list/remove も引き続きアプリ独自のファイル探索・削除で処理する。
   - 不採用理由: `skills` 側の管理状態と乖離しやすく、将来互換性と追跡性を下げる。

## Risks / Trade-offs

- **[Risk] `skills` ライブラリの将来仕様変更で一覧/削除連携が破綻する**
  - Mitigation: 連携点を capability requirements とテストケースで固定し、アップデート時に差分確認を必須化する。

- **[Risk] symlink と実体パスの混在で重複検出や削除対象判定が誤る**
  - Mitigation: realpath ベースの正規化ルールを仕様化し、循環リンク回避をテストで担保する。

- **[Risk] セッション再初期化タイミングによって直後の UX が不安定になる**
  - Mitigation: 反映規約を「次回送信」に統一し、UI 文言と実装挙動を一致させる。

- **[Trade-off] `skills` ライブラリ寄せによりアプリ独自最適化の自由度が下がる**
  - Mitigation: アプリ独自処理は UI/IPC 層に限定し、管理データ面はライブラリ準拠で安定性を取る。
