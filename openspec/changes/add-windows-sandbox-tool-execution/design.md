## Context

現在のツール実行はホスト OS 上で直接行われるため、Windows ではコマンドの副作用がローカル環境へ及ぶ。今回の変更では、設定画面で Windows Sandbox 実行を選択できるようにし、ON 時のみ隔離環境を起動してツールを実行する。参考実装として `codex-rs/windows-sandbox-rs` の流れを採用するが、実装は最小限（起動、最小セットアップ、コマンド投入、結果回収）に限定する。

制約:
- Windows 以外の OS の挙動は変更しない。
- 既存のホスト実行フローとの互換性を維持する。
- Sandbox 初回起動コストは許容し、設計の単純さを優先する。

## Goals / Non-Goals

**Goals:**
- 設定画面で Windows Sandbox 実行の ON/OFF を切り替え可能にする。
- ON 時にのみ Sandbox 実行経路へ分岐し、最小限の実行環境を自動構築する。
- OFF 時は既存のホスト実行を維持し、既存利用者の挙動を壊さない。
- 実行失敗時に、設定値と分岐経路が追跡可能なログを残す。

**Non-Goals:**
- Linux/macOS 向けの隔離実行の追加。
- 高度なリソース制御（CPU/メモリ制限の詳細チューニング）。
- Sandbox 内キャッシュの永続化最適化。
- 既存ツールの仕様変更（Bash/Write の API 自体は維持）。

## Decisions

1. 設定モデルに `useWindowsSandboxForTools`（仮名、boolean）を追加する
- 理由: 既存設定と同様に永続化し、UI と実行系で同一フラグを参照できるようにする。
- 代替案: 環境変数だけで切り替える。
  - 不採用理由: ユーザー操作性が低く、設定画面での明示的切替という要件を満たせない。

2. Main 側に `ToolExecutionModeResolver` を設け、`host` / `windows-sandbox` を実行直前に判定する
- 理由: 実行分岐を 1 箇所に閉じ、各ツール実装への影響を最小化する。
- 代替案: 各ツール内で個別に設定判定する。
  - 不採用理由: 重複実装が増え、分岐漏れのリスクが高い。

3. Sandbox 実行は専用アダプタ（`WindowsSandboxExecutor`）に集約する
- 理由: `windows-sandbox-rs` の責務分離（起動/投入/回収）を踏襲しつつ、アプリ本体は抽象インターフェースで扱える。
- 代替案: 既存 shell 実行モジュールに直接 Sandbox 処理を埋め込む。
  - 不採用理由: 既存ホスト実行ロジックが複雑化し、回帰リスクが上がる。

4. Sandbox の準備は「ON 時に必要最小限を都度構築」方式にする
- 理由: 先に重い常駐管理を入れず、最短で価値を届ける。
- 代替案: 起動時に常時 Sandbox をウォームアップ。
  - 不採用理由: OFF ユーザーにもコストが発生し、最小実装方針に反する。

## Risks / Trade-offs

- [Sandbox 起動が遅く体感が悪化] → 実行中ステータス表示とログで待機を明示し、将来のウォームアップ導入余地を残す。
- [Windows 機能無効環境で失敗] → 設定 ON 時に事前チェックし、無効時は具体的な有効化ガイドを返す。
- [Sandbox 内セットアップ失敗] → 失敗点（起動/投入/実行/回収）を分類してエラー返却し、ホスト実行へ自動フォールバックは行わない（安全性優先）。
- [運用中のデバッグ難度上昇] → 実行モード、設定値、Sandbox セッション識別子を構造化ログに残す。

## Migration Plan

1. 設定スキーマにフラグを追加し、既存ユーザーはデフォルト OFF とする。
2. 設定画面にトグルを追加し、変更時に即時保存する。
3. Main のツール実行入口にモード判定と Sandbox 実行アダプタを接続する。
4. Windows で ON/OFF の手動確認と既存回帰確認を行う。
