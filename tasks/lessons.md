## 2026-03-22 (addendum)

| Mistake/Context | Why it mattered | Rule to repeat |
|---|---|---|
| I added the Windows dictation path and checked helper availability before checking whether the platform itself was supported. | That made unsupported platforms like Linux return `SERVICE_UNAVAILABLE` instead of `UNSUPPORTED_PLATFORM`, which weakens tests and user-facing error semantics. | For every new OS branch: decide `supported vs unsupported` first, then check helper/dependency availability, then run the platform-specific implementation. Do not collapse unsupported and unavailable into the same guard. |
| I reused the browser capture sample rate as-is for Windows speech recognition. | `System.Speech` is much more sensitive to input format than the macOS path, so feeding arbitrary browser sample rates degrades recognition quality even when the pipeline is otherwise correct. | When adding speech features across platforms, normalize captured audio to the recognizer-friendly format first. For Windows `System.Speech`, prefer mono 16-bit PCM at `16kHz` unless there is a verified reason not to. |
| I initially tried to improve Windows dictation by tuning the old `System.Speech` file-based path instead of switching the engine. | Once the recognizer itself is the limiting factor, input cleanup only buys small gains and can waste time while the user still gets poor results. | For speech-quality bugs, separate "pipeline bug" from "engine quality ceiling" early. If the ceiling is the issue, switch to the better native engine before spending more time on preprocessing. |
| I wrote a PowerShell helper script with non-ASCII status strings and it became unreadable enough to break parsing/execution in this environment. | Utility scripts fail in frustrating ways when shell encoding and file encoding assumptions do not match, especially on Windows PowerShell. | Default new automation/install scripts to ASCII-only messages unless Unicode is required and already proven safe in that script path. |
| I treated “C++ Build Tools installed” as equivalent to “all Rust MSVC targets are buildable” on Windows ARM. | Windows on ARM needs target-matching MSVC toolchains and libraries, so having x64/x86 tools does not prove ARM64 Rust native builds will link. | On Windows native builds, verify the exact target toolchain artifacts you need (`bin/.../link.exe` and `lib/<arch>/vcruntime.lib`) before assuming the environment is ready. |
| I added an x64 fallback path for Windows ARM without first standardizing where the built binaries live. | Manual Cargo builds can succeed while the app still fails at runtime if development lookup and packaging still point at the old target-specific paths. | When native builds may emit different target layouts, always copy the final executable into a stable `bin/` directory and make runtime and packaging resolve that path first. |
| I let a Windows speech privacy prerequisite surface as a raw system exception string. | The feature was technically working, but the user still hit a dead end because the app did not explain which Windows setting blocked speech recognition. | For OS-gated features, map known platform error strings to actionable product messages that name the exact Settings page or prerequisite the user needs to enable. |
| I shipped Windows dictation on top of one-shot recognition even though the UI was press-and-hold and users naturally dictated longer phrases. | A push-to-talk UX implies explicit start/stop control, and one-shot recognition will often finalize too early on silence or segmentation. | When a speech feature uses press-to-start and release-to-finish semantics, prefer continuous recognition and send an explicit stop signal instead of relying on one-shot recognition to guess the utterance boundary. |
| I added new native build outputs without immediately teaching Git which paths were generated artifacts. | Fresh helper binaries and target directories created noisy status output and made it harder to see the real source changes that mattered. | Whenever a task introduces a new build output directory, update `.gitignore` in the same turn so generated artifacts never linger as accidental untracked changes. |
